初识redis

引言：写点有意思的redis

1. 简单介绍

Redis is an open source，in-memory data structure store，used as a database，cache and message broker.

Redis 是一个开源的，基于内存的数据结构存储，可以用作数据库、缓存和消息中间件。

“Redis is written in ANSI C”–>Redis由C语言编写

Redis 是基于内存，常用于缓存的一种技术，并且redis存储的方式是以 key-value 的形式。

为什么要用 Redis 而不是 map？

  • key-value 不就是 Map，java 实现的 Map 是本地缓存，如果有多个机器的话，每个实例都需要保存一份本地缓存，缓存具有不一致性。
  • Redis 实现的是分布式缓存，如果有多台实例的话，每个实例都会共享一份缓存，缓存具有一致性。
  • java 的 Map 不是专业做缓存的，JVM 的内存太大容易挂掉。一般用做于容器来存储临时数据，缓存的数据随着 JVM 的销毁而结束。Map 所存储的数据结构，缓存机制等等都是需要程序员自己定义的。
  • Redis 是专业做缓存的，可以用几十个 G 内存来做缓存。Redis 一般用作于缓存，可以将缓存的数据保存在硬盘里，Redis 重启之后就可以将其恢复。原生提供丰富的数据结构、缓存过期机制等等很多简单好用的功能。

为什么要用缓存？

  • 网站访问很慢，一般是数据库支撑不住。因为一般数据库的读写都要经过磁盘，读磁盘的速度很慢（相对于读内存来说）
  • mybatis、hibernate 不仅仅有一级缓存还有二级缓存这样的功能（终究来说还是本地缓存）。目的就是为了减少数据库的读写
  • 使用缓存可以提高性能（缓存的查询比数据库查询快，内存和硬盘的速度不在一个量级）、提高并发能力，缓存会分担一部分请求，可以支持更高的并发

2. Redis 数据结构

常用的数据结构：String list hash set sortset

Redis 是由 key-value 存储的，Redis 的 key 一定是字符串，value 可以是上面的类型

Redis 并没有直接使用这些数据类型来存储键值，而是为键创建键对象，为值创建值对象。只不过键对象中的类型为 String

2.1. SDS 简单动态字符串

Redis 中的字符串跟 C 语言中的字符串，是有点差距的。

Redis 使用 sdshdr 结构来表示一个SDS值：

使用SDS的好处：

  • sdshdr 数据结构中用 len 属性记录了字符串的长度。那么获取字符串长度时，时间复杂度只需要 O(1)
  • SDS 不会出现溢出的问题，如果修改 SDS 时，空间不足。会先扩展空间，再进行修改！（内部实现了动态扩展机制）
  • SDS 减少内存分配次数（空间预分配机制）。在扩展空间时，除了分配修改时所必要的空间，还会分配额外的空闲空间（free 属性）
  • SDS 是二进制安全的，所有的 SDS API 都会以处理二进制的方式来处理 SDS 存放在 buf 数组里的数据

2.2. 链表

  • 无环双向链表
  • 获取表头指针，表尾指针，链表节点长度的时间复杂度均为O(1)
  • 链表使用 void* 指针来保存节点值，可以保存各种不同类型的值

2.3. 哈希表

字典其实就是在哈希表上再做了一层抽象

在 Redis 中，key-value 的数据结构底层就是哈希表实现的。在 java 中哈希表其实就是数组+链表的形式来构建的。

Redis 为了更好的使用，对哈希表进行了进一步的封装。

Redis 中的哈希算法和哈希冲突跟 java 的差不多，他们的差异：

  • Redis 哈希冲突时，是将最新的节点添加到链表的表头
  • JDK 1.8 之后，java 在解决哈希冲突的时候，是将新的节点添加到链表的表尾

2.4. rehash 的过程

Redis 专门用一个哈希表来进行 rehash。这和 java 的一次性直接 rehash 是有区别的。

在对哈希表进行扩容和收缩的过程中，rehash 过程并不是一次性完成的，而是渐进式的完成的。

Redis 在 rehash 的时候采用渐进式是因为：数据量过大的话，一次性 rehash 会有庞大的计算量，这可能会让服务器一段时间停止服务

Redis具体是rehash时这么干的：

  • (1:在字典中维持一个索引计数器变量rehashidx，并将设置为0，表示rehash开始。
  • (2:在rehash期间每次对字典进行增加、查询、删除和更新操作时，除了执行指定命令外；还会将ht[0]中rehashidx索引上的值**rehash到ht[1]**，操作完成后rehashidx+1。
  • (3:字典操作不断执行，最终在某个时间点，所有的键值对完成rehash，这时将rehashidx设置为-1，表示rehash完成
  • (4:在渐进式rehash过程中，字典会同时使用两个哈希表ht[0]和ht[1]，所有的更新、删除、查找操作也会在两个哈希表进行。例如要查找一个键的话，服务器会优先查找ht[0]，如果不存在，再查找ht[1]，诸如此类。此外当执行新增操作时，新的键值对**一律保存到ht[1]**，不再对ht[0]进行任何操作，以保证ht[0]的键值对数量只减不增，直至变为空表。

2.5. 跳跃表

跳跃表（shiplist）是实现 sortset（有序集合）的底层数据结构之一。

什么是跳跃表？

  • 跳跃表是一种基于有序链表的扩展，简称调表
  • 其实就是使用关键节点作为索引的一种结构

Redis 的跳跃表实现由 zskiplist 和 zskiplistNode 两个结构组成。其中 axkiplist 保存跳跃表的信息（表头 表尾节点 长度），zskiplistNode 则表示跳跃表的节点。

2.6. 整数集合（intset）

整数集合是 set （集合）的底层数据结构之一。当一个 set （集合）只包含整数值元素，并且元素的数量不多时，Redis 就会采用整数集合（intset）作为 set （集合）底层实现。

整数集合保证了元素是不会出现重复的，并且时有序的（从小到大排序）

2.7. 压缩列表（ziplist）

压缩列表（ziplist）时 list 和 hash 的底层实现之一。如果 list 的每个都是小整数值，或者是比较短的字符串，压缩列表（ziplist）作为 list 的底层实现。

压缩列表（ziplist）是 Redis 为了节约内存而开发的，是由一系列的特殊编码的连续内存块组成的顺序性数据结构。

2.8. 字符串对象

String 类型有三种编码格式：

  • int：整数值，这个整数值可以使用 long 类型来表示
    • 如果是浮点数，那就用 embstr 或者 raw 编码。具体用哪个就看这个数的长度了
  • embstr：字符串值，这个字符串值的长度小于32字节
  • raw：字符串值，这个字符串值的长度大于32字节

embstr 和 raw 的区别：

  • raw 分配内存和释放内存的次数是两次，embstr 是一次
  • embstr 编码的数据保存在一块连续的内存里面

编码之间的转换：

  • int 类型如果存的不再是一个整数值，则会从 int 转成 raw
  • embstr 是只读的，在修改的时候会从 embstr 转成 raw
