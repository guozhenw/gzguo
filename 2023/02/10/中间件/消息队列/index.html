初始消息队列

引言：消息队列的基础内容，入门理解。

1. 消息队列

消息队列（Message Queue）：把传递的信息存在队列中

队列（Queue）：一种先进先出的数据结构

目前在 java 里，已经实现了不少的队列了！那为什么还要消息队列这种中间件呢？这就好像 java 有了 map，但是我们还需要 Redis！

所以为什么有了队列还要有消息队列？

在 java 中实现的队列为什么不能满足我们的需求？java 中的队列只是针对内存的，我们现在所说的消息队列针对的是多个服务之间的。

黑话：

  • 生产者：往队列里面放东西的（放数据）
  • 消费者：取队列中的东西

1.1. 消息队列的作用

1.1.1. 解耦

前提：A 中独有的一份数据

过程：

  • 第一天可能有 B、C、D 需要，那么我们的类中就要把 A 中的数据传给 B、C、D。
  • 第二天 B 不要了，E 又需要了
  • …

结果：我们每天都得改代码，这样的话，还需要专人维护，很烦

问题出现了，消息队列来解决

有了消息队列，A 只要把信息写到消息队列中，谁想要谁自己取。这样和我 A 就没关系了。A 解耦了！

1.1.2. 异步

原先 A 中的数据，B、C、D 都要。

顺序执行的话，总的时间就是 B+C+D

放大消息队列中，异步之后，执行时间可能就是固定的 100ms，会有很大的效率提升

1.1.3. 削峰/限流

系统最大承载的请求量是 3000；现在一次性来了 5000，系统直接崩溃了！

加一个消息队列，把请求放到消息队列里边，服务自己根据自己的处理能力去取请求，比如可以一次取两千、再取两千、最后取一千，成功处理完成。这样就可以避免系统崩溃。

理论上来说，一次性取多少是可以由我们配置的！

感觉这里消息队列的唯一好处就是可以排个序，先进来的用户请求可以先执行

1.2. 使用消息队列会带来的很多问题

java 中的队列/JDK实现的队列，虽然有很多种类，但都是简单的内存队列。

下面我们看看消息队列可能要考虑的问题！

1.2.1. 高可用

无论是我们使用消息队列来做解耦、异步还是削峰，消息队列肯定不能是单机的。试想一下，如果是单机消息队列，万一这台机器挂了，那我们整个系统就几乎不可用了。

所以当我们项目中使用消息队列，都是集群/分布式的。要做集群/分布式就必然希望该消息队列能够提供现成的支持，而不是自己手写代码去实现。

1.2.2. 数据丢失问题

我们将数据写到消息队列上，系统 B 和 C 还没来得及取消息队列的数据，就挂掉了。如果没有任何的措施，我们的数据就丢了。怎么解决这种情况呢？

学过 Redis 的都知道，Redis 可以将数据持久化到磁盘上，万一 Redis 挂了，还能从磁盘将数据恢复过来。

同样的，消息队列中的数据也需要存在别的地方，这样才能尽可能减少数据的丢失。

那么问题来了，消息队列中的数据要存在哪呢？

  • 磁盘
  • 数据库
  • Redis
  • 分布式文件系统

同步存储还是异步存储？

1.2.3. 消费者怎么得到消息队列的数据？

消费者怎么从消息队列里边得到数据？有两种办法：

  • 生产者将数据放到消息队列中，消息队列有数据了，主动叫消费者去拿（俗称 push）
  • 消费者不断去轮询消息队列，看看有没有新的数据，如果有就消费（俗称 pull）

1.2.4. 其他

除了这些，我们在使用的时候还得考虑各种的问题：

  • 消息重复消费了怎么办啊？
  • 我想保证消息是绝对有顺序的怎么做？
  • ……..

虽然消息队列给我们带来了那么多的好处，但同时我们发现引入消息队列也会提高系统的复杂性。市面上现在已经有不少消息队列轮子了，每种消息队列都有自己的特点，选取哪种MQ还得好好斟酌。

1.3. 最后
