1.1. 分布式

集群是什么？

我有一个商城网站，开发完成之后部署到了 Linux 服务其上。突然有一天访问人数大幅度增加，网站变得很慢。我该怎么办？

  • 给 Linux 服务器增加配置，增加内存和外存。

又过了一段时间，访问网站的人更多了，但是现在 Linux 服务器已经是顶配了，我又该怎么办？

  • 再买一台 Linux 服务器，把我的项目同时部署到两台服务器上。在用户访问的时候，我把请求做一个分流，这样两台服务器就可以互相分担访问请求太多的压力了！

分布式是什么？

分布式是我开发了一个项目，在部署的时候，我把一个项目根据服务的不同拆开。把各个服务部署到不同的 Linux 服务器上，这些服务通过各自的网络协议互相访问。

  • 分不同的服务有什么意义呢？

问得好，首先，我们肯定有访问量高的服务，相对而言也有访问量比较低的服务，这个时候，通过我们的分析，我们可以把访问量高的服务放在好一点的服务器上，从而可以提高我们的效率。不仅仅如此，通过功能拆分，我们的模块之间相互独立，这样可以便于扩展、复用性高。

集群和分布式可以同时使用吗？

集群和分布式并不冲突，可以有分布式集群

相当于我的商城项目分成了销售服务、首页轮播服务、购物车服务。分服务就是分布式。

现在我把销售服务搞了2份到两个服务器上，这就是集群。

上面的这种情况就是分布式集群。

分布式、微服务、SOA有什么区别，有什么相同点？

1.1.1. 分布式–CAP理论

什么是 CAP？

  • C，consistency：数据一致性

    • 所有的节点都有最新的数据
      • 这里的节点就是分布式分出来的某个服务

  • A，availability：可用性

    • 数据具备高可用性

  • P，partition-tolerance：分区容忍性

    • 容忍网络出现分区，分区之间的网络不可达

      • 什么是分区？

        分区就是出现网络问题之后，节点之间不能联通，出现断口，把原先完整的网络分成了几个区域

      • 出现分区之后，我们是否允许访问的请求继续成功？

        • 如果允许继续访问，那么 c 发生的变化传不到 a。这个时候我们就是选择了可用性（availability），抛弃了数据一致性（consitency）
        • 如果不允许继续访问，要等到 a 和 c 之间恢复通信之后才可以继续访问。这个时候我们就选择了数据一致性（consitency），放弃了可用性（aivilability）

1.1.2. 分布式–CAP理论梳理

一般来说，CAP 中的 P 是不可或缺的。P（分区容忍性 partition-tolerance），这个网络问题是客观存在的，也就是说分区问题是客观存在的，我们要保证出现分区的时候也可以正常运行。

CAP 是无法兼顾的，从上面的例子也可以看出，我们可以选 AP，也可以选 CP。但是，要注意的是：不是说选了 AP，C 就完全抛弃了。不是说选了 CP，A 就完全抛弃了

在 CAP 理论中，C 所代表的一致性是强一致性（每一个节点必须是最新版本），其实一致性还有其他的级别：

  • 弱一致性：弱一致性是相对于强一致性而言，它不保证总能得到最新的值；
  • 最终一致性（eventual consistency）：放宽了对事件的要求，在被调完成操作响应后的某个时间点，被调多个节点的数据最终达成一致。

A 所代表的一致性也可以定义成 0-100 的连续区间

  • 容错可用性：每年可容忍停机1min
  • 极高可用性：~~5min
  • 具有故障自动恢复能力的可用性：~~53min
  • 高可用性：~~8.8h
  • 商品可用性：~~43.8min

所以，CAP 理论定义的其实是在容忍网络分区的条件下，“强一致性”和“极致可用性”无法同时达到

1.2. SpringCloud 就是这么简单

单从概念来说，是非常容易理解的。下面来熟悉一下 SpringCloud 最基础的知识。

1.2.1. 为什么需要 SpringCloud？

前面提到了分布式，我们要把自己的项目拆开不同的服务。但是拆会出现各种各样的问题，而 SpringCloud 提供了一整套的解决方案！

SpringCloud 的基本功能：

  • 服务治理：SpringCloud Eureka
  • 客户端负载均衡：SpringCloud Ribbon
  • 服务器容错保护：SpringCloud Hystrix
  • 声明式服务调用：SpringCloud Zuul
  • API 网关服务：SpringCloud Zuul
  • 分布式配置中心：SpringCloud Config

SpringCloud 的高级功能：

  • 消息总线：SpringCloud Bus
  • 消息驱动的微服务：SpringCloud Stream
  • 分布式服务跟踪：SpringCloud Sleuth

1.2.2. 引出 Eureka

现在我们把项目分成了很多的服务，每个服务之间的通讯要怎么实现？

子系统之间并不在一个环境下，需要远程调用。远程调用可能会用到 httpClient、WebService 等等这些技术。

在远程调用的时候，我们可以直接用 ip 地址调用。

直接用 ip 地址调用有一个小问题？有一天我们的 ip 地址变了怎么办？

为了解决微服务架构中的服务实例维护问题（ip 地址），产生了大量的服务治理框架和产品。这些框架和产品的实现都围绕着服务注册与服务发现机制来完成对微服务实例的自动化管理。

在 SpringCloud 中我们的服务治理框架一般使用的就是 Eureka

Eureka 怎么解决 ip 地址会发生变化的问题？

创建一个新的服务 Server#，将之前项目中分出来的服务都注册到新服务 Server#上，用新服务 Server#来维护这些已经注册进来的信息。

项目中分出来的服务只需要取 Server#上的注册清单（注册清单上写着服务名和对应的服务 ip 地址），分出来的服务不再调用具体的 ip，而是调用清单上的服务名。

1.2.3. Eureka 细节

Eureka 专门用于给其他服务注册的那个服务称为 Eureka Server（服务注册中心），其余注册到 Eureka Server 的服务叫 Eureka Client

在 Eureka Server 一般我们会这样配置：

Eureka Client 分为服务提供者和服务消费者

  • 但很有可能，某个服务既是服务提供者，又是服务消费者

如果在网上看到 SpringCloud 的某个服务配置没有“注册”到 Eureka-Server 也不用过于惊讶（但是它可以获取 Eureka 服务清单）

  • 很可能只是作者把该服务认作为单纯的服务消费者，单纯的服务消费者是无需对外提供服务的，所以也就不需要出现在服务清单上了，也就无需注册到 Eureka Server 中了

Eureka 治理机制：

  • 服务提供者
    • 服务注册：启动的时候会通过发送 REST 请求的方式将自己注册到 Eureka Server 服务上，同时带上了自身服务的一些元数据信息
    • 服务续约：在注册完服务后，服务提供者会维护一个心跳用来持续告诉 Eureka Server“我还活着”
    • 服务下线：当服务实例进行正常的关闭操作时，它会触发一个服务下线的 REST 请求给 Eureka Server，告诉服务注册中心“我要下线了”
  • 服务消费者
    • 获取服务：当我们启动服务消费者的时候，它会发送一个 REST 请求给服务注册中心，来获取上面注册的服务清单
    • 服务调用：服务消费者在获取到服务清单后，通过服务名可以获得具体提供服务的实例名和该实例的元数据信息。在进行服务调用的时候，优先访问同处一个 Zone 中的服务提供方
  • Eureka Server（服务注册中心）
    • 失效剔除：默认每隔一段时间（默认为60秒）将从当前清单中超时（默认为90秒）没有续约的服务剔除出去
    • 自我保护：Eureka Server 运行期间，会统计心跳失败的比例，在15分钟之内是否低于85%（通常由于网络不稳定导致）。Eureka Server会将当前的实例注册信息保护起来，让这些实例不会过期，尽可能保护这些注册信息。

1.2.4. 引出 RestTemplate 和 Ribbon

通过 Eureka 服务治理框架，我们可以通过服务名来获取具体的服务实例位置了（IP）。一般使用 SpringCloud 的时候不需要自己手动创建 HttpClient 来进行远程调用。

可以使用 Spring 封装好的 RestTemplate 工具类，使用起来很简单。

为了实现服务的高可用，我们可以将服务提供者集群。集群的意思就是多搞几台服务器，上面部署同样的服务！

现在想要将上面部署的多个相同服务合理分摊用户的请求（专业来说就是 负载均衡），可能你想到的会是 nginx；

其实 SpringCloud 也支持负载均衡的功能，只不过它是客户端的负载均衡，这个功能实现就是 Ribbon！

负载均衡又区分了两种类型：

  • 客户端负载均衡（Ribbon）
    • 服务实例的清单在客户端，客户端进行负载均衡算法分配
    • （从上面我们已经知道了：客户端可以从 Eureka Server 中得到一份服务清单，在发送请求的时候通过负载均衡算法，在多个服务器之间选择一个进行访问）
  • 服务端负载均衡（Nginx）
    • 服务实例的清单在服务端，服务器进行负载均衡算法分配

1.2.5. Ribbon 细节

Ribbon 是支持负载均衡，默认的负载均衡策略是轮询，我们也是可以根据自己的实际需求来自定义负载均衡策略的

实现起来也很简单，继承 AbstractLoadBalancerRule 类，重写 public Server choose（ILoadBalancer 1b， Object key）即可

SpringCloud 在 CAP 理论是选择了 AP 的，在 Ribbon 中还可以配置重试机制的（有兴趣的可以去看看）

1.2.6. 引出 Hystrix

到目前为止，我们的服务看起来已经很好了。我们有分布式服务，可以用 Eureka 来进行优化，避免直接写死 IP。真正写代码调用的时候还有 RestTemplate 来帮助我们解析服务名为 ip；我们还有集群，我们的集群可以交给 Ribbon 来进行负载均衡（默认轮询、客户端）

但是，如果我们在调用多个远程服务时，某个服务出现延迟，会怎么样？？

在高并发的情况下，由于单个服务的延迟，可能导致所有的请求都处于延迟状态，甚至在几秒钟就使服务处于负载饱和的状态，资源耗尽，直到不可用，最终导致这个分布式系统都不可用，这就是“雪崩”

针对上诉的问题，SpringCloud Hystrix 实现了短路器、线程隔离等一系列服务保护功能。

  • Fallback（失败快速返回）：当某个服务单元发生故障（类似用电器发生短路）之后，通过断路器的故障监控（类似熔断保险丝），向调用方返回一个错误响应，而不是长时间的等待。这样就不会使得线程因调用故障服务被长时间占用不释放，避免了故障在分布式系统中的蔓延。
  • 资源/依赖隔离（线程池隔离）：它为每一个依赖服务创建一个独立的线程池，这样就算某个依赖服务出现延迟高的情况，也只是对该依赖服务的调用产生影响，而不会拖慢到其他服务

Hystrix 提供了几个熔断的关键参数：滑动窗口大小（20）、熔断器开关间隔（5s）、错误率（50%）

  • 每当 20 个请求中，有 50% 失败时，熔断器就会打开，此时再调用此服务，将会直接返回失败，不再调用远程服务。
  • 知道5s钟之后，重新检测该触发条件，判断是否把熔断器关闭，或者继续打开

Hystrix 还有请求合并、请求缓存这样强大的功能，在此我就不具体说明了，有兴趣的同学可以继续深入学习。

1.2.7. Hystrix 仪表盘

Hystrix 仪表盘：主要用来实时监控 Hystrix 的各项指标信息。通过 Hystrix Dashboard 反馈的实时信息，可以帮助我们快速发现系统中存在的问题，从而即使采取应对措施。相当于一个预言家。

除了可以开启单个实例的监控页面之外，还有一个监控端点 /turbine,stream是对集群使用的。从端点的命名中们可以引入 Turbine，通过它来汇集监控信息，并将聚合后的信息提供给 HystrixDashboard 来集中展示和监控。

1.2.9. 引出 Zuul

目前还有两个比较麻烦的问题：

  1. 路由规则与服务实例的维护问题：外层的负载均衡（nginx）需要维护所有的服务实例清单
  2. 签名校验、登录校验冗余问题：为了保证对外服务的安全性，我们在服务端实现的微服务接口，往往都会有一定的权限校验机制，但是我们的服务是独立的，我们不得不在这些应用中都是先这样一套校验逻辑，这就会造成校验逻辑的冗余。

SpringCloud Zuul 解决这两个问题的方法：

  • SpringCloud Zuul 通过与 SpringCloud Eureka 进行整合，将自身注册为服务治理下的应用，同时从 Eureka 中获得了所有其他微服务的实例信息。外层调用都必须通过 API 网关，使得将维护服务实例的工作交给了服务治理框架自动完成。
  • 在 API 网关服务上进行统一调用来对微服务接口做前置过滤，以实现对微服务接口的拦截和校验

Zuul 天生就拥有线程隔离和断路器的自我保护功能，以及对服务调用的客户端负载均衡功能，也就是说：Zuul也是支持 Hystrix 和 Ribbon。

关于 Zuul 还有很多知识点：

  • 路由匹配（动态路由）
  • 过滤器实现（动态过滤器）
  • 默认会过滤掉 Cookie 与敏感的 HTTP 头信息（额外配置）

1.2.10. 对 Zuul 的疑问

Zuul 支持 Ribbon 和 Hystrix，也能够实现客户端的负载均衡。我们的 Feign 不也是实现客户端的负载均衡和 Hystrix 的吗？既然 Zuul 已经能够实现了，那我们的 Feign 还有必要吗？

可以这样理解：

  • Zuul 是对外暴露了唯一接口相当于路由的是 controller 的请求，而 Ribbon 和 Feign 路由了 service 请求。
  • Zuul 做最外层请求的负载均衡，而 Ribbon 和 Feign 做的是系统内部各个微服务的 service 的调用的负载均衡

有了 Zuul，还需要 Nginx 吗？他两可以一起使用吗？

  • 我的理解：Zuul 和 Nginx 可以一起使用的（毕竟我们的 Zuul 也是可以搭成集群来实现高可用的），要不要一起使用得看架构的复杂程度了（业务）

1.2.11. 引出 SpringCloud Config

随着业务的扩展，我们的服务会越来越多，每个服务都有自己的配置文件。

既然是配置文件，给我们的配置难免会有改动，比如说密码，一旦改动，所有的服务都得修改配置文件！

在分布式系统中，某一个基础服务信息变更，都很可能会引起一系列的更新和重启！

SpringCloud Config 项目是一个解决分布式系统的配置管理方案。它包含了 Client 和 Server 两个部分，server 提供配置文件的存储、以接口的形式将配置文件的内容提供出去，client 通过接口获取数据、并依据此数据初始化自己的应用

  • 简单来说，使用 SpringCloud Config 就是将配置文件放到统一的位置管理（比如 GitHub），客户端通过接口去获取这些配置文件

  • 在 GitHub 上修改某个配置文件，应用加载的就是修改后的配置文件。

  • SpringCloud Config其他的知识：

    • 在SpringCloud Config的服务端， 对于配置仓库的默认实现采用了Git，我们也可以配置SVN。
    • 配置文件内的信息加密和解密
